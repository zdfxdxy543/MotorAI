**第二轮 Agent 应该读取什么**

其中最重要的是：

`evaluation_result.json`：看具体哪些指标没达标。  
`optimization_history.jsonl`：看参数是怎么变的，哪些参数方向有效，哪些调参陷入无效。  
`controller_loop_ids_generated.json`：看这次到底生成了什么控制结构。  
`paras.generated.h`：看这个结构暴露了哪些可调参数，以及最终参数能否迁移到下一轮。

**现在的 design_strategy 和 controller_loop_ids_generated 足够吗**

不够，但它们是必要输入。

`design_strategy.json` 现在更像“生成意图”：这个 candidate 想偏稳健、偏快速、偏抗扰。  
`controller_loop_ids_generated.json` 是“结构选择结果”：比如选了 current loop、mech loop、pid/mit/smc 等。

但它们缺几个关键东西：

1. **生成后的结构摘要**
   不只是 selected loops，还要有结构签名、控制方法、环路层级、参数组。

2. **参数初值和最终参数**
   第二轮需要知道第一轮哪些参数收敛到了哪里。

3. **失败指标和失败程度**
   例如超调差多少、调节时间差多少、稳态误差差多少。

4. **结构和指标之间的关系**
   例如：快速响应 candidate 分数高但超调超标，说明第二轮可以保留较快外环，但要加强限幅/滤波/抗饱和。

所以我建议新增一个文件：

```text
candidate_xx/log/generate/controller_manifest.json
```

它由 generate 阶段写出，内容可以简洁但必须结构化：

```json

  "schema_version": 1,

  "structure_signature": "current_loop+mech_loop:pid",
  "loop_hierarchy": ["current_loop", "mech_loop"],
  "control_methods": ["pid"],
  "selected_loops": [],

    "ctl_main_c": "src/ctl_main.c",
    "paras_header": "src/paras.generated.h"
  },
  "tunable_parameters": [    {
      "name": "MOTORAI_SPEED_KP",

    "preferred_control_methods": ["pid", "gain_scheduling"]
  }
}
```

`controller_loop_ids_generated.json` 偏 LLM 输出，`controller_manifest.json` 偏系统内部证据。后者给第二轮 agent 用，会稳很多。

**目录结构怎么放**

我不建议在同一个 `candidate_01` 下面再分结构迭代轮次。因为 candidate 本质上应该代表“一次具体结构假设 + 对应源码 + 对应调参结果”。如果在 candidate 下放 round，会导致 `src/ctl_main.c`、`paras.generated.h`、optimize log 都变成可变状态，后面追溯会很麻烦。

更推荐这样：

```text
project/

    round_01/
      candidates/
        candidate_01/
        candidate_02/
      round_result.json
      round_feedback.json

    round_02/
      candidates/
        candidate_01/
        candidate_02/
      input_feedback.json
      round_result.json
      round_feedback.json
```

如果为了兼容现有代码暂时不想大改，也可以折中命名：

```text
project/
  candidates/
    round_01_candidate_01/
    round_01_candidate_02/
    round_02_candidate_01/
    round_02_candidate_02/
```

但从长期设计看，`rounds/round_xx/candidates/candidate_xx` 更干净。

1. 用户原始需求
2. 当前支持的结构空间
3. 第一轮 `round_feedback.json`
